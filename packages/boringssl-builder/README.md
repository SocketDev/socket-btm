# boringssl-builder

Builds BoringSSL as a private static library for embedding inside
`node:smol-http`. The prefix `BORINGSSL_PREFIX=smol` rewrites every
public symbol (`SSL_CTX_new` → `smol_SSL_CTX_new`, etc.) so the
resulting `libsmol_crypto.a` / `libsmol_ssl.a` link cleanly alongside
Node's bundled OpenSSL with zero collisions.

## Why a separate builder

BoringSSL takes 7–10 minutes per platform to compile cold. `node-smol`
runs 8-platform CI; doing this inline would burn ~80 minutes per cache
miss. The fleet pattern (curl-builder, lief-builder) is to prebuild
once, publish as a stub (`@socketbin/boringssl-<platform-arch>`), and
have downstream consumers pull the prebuilt artifact.

## Upstream pin

We pin to `oven-sh/boringssl` (BoringSSL + Electron patches) at the
same SHA Bun ships in `bun-v1.3.14`:

- Repo: `oven-sh/boringssl`
- SHA: `0c5fce43b7ed5eb6001487ee48ac65766f5ddcd1`

The fork tracks BoringSSL `master` with Electron's compat patches
already applied — keeps us aligned with two heavily-tested downstream
consumers (Bun + Electron) instead of rolling our own patches.

## Bumping the pin

1. Find Bun's current pin: `cat upstream/bun/scripts/build/deps/boringssl.ts` (look for `BORINGSSL_COMMIT`).
2. Update the submodule SHA in `.gitmodules` + bump the version comment.
3. `pnpm run build` to validate.
4. Bump `boringssl` cache version in `.github/cache-versions.json`.

## Outputs

Per-platform artifacts:

```
build/<mode>/<platform-arch>/out/Final/
├── lib/
│   ├── libsmol_crypto.a
│   └── libsmol_ssl.a
└── include/
    └── openssl/          # prefixed headers (generated via boringssl/util/make_prefix_headers.go)
        ├── ssl.h
        ├── crypto.h
        └── ...
```

Consumed by `lsquic-infra` and `node-smol-builder`'s
`additions/source-patched/deps/boringssl.gypi` (auto-generated by
`prepare-external-sources.mts`).
